What I did

I worked as a software engineer for about 5 years, mostly in the web development space.

I started out as a full-stack developer, building a web application for ecologists to visualize simulated pollen distribution. I was able to create a pipeline with C++, PHP, HTML and Javascript that allowed users to geographically plot maize fields and visualize the pollen distribution in a web browser. I built on a proprietary simulation engine for low density particles called AUSTAL and created a web interface for it.

This was my first experience with web development and I loved it. My biggest lesson back then: Don't cook your own TIFF images, there are libraries for that.

What I build

I like to build things. I enjoy the process more than the end result quite a lot. Recently I started and ditched over twenty iterations of a web app to explore the landscape of web development. I compared different languages, frameworks, libraries, and tools. I tried out different approaches to state management, routing, server side and client side rendering, islands architecture, MVC patterns and more. Here are some of the things I learned:

  • There is allmost no reason to use Javascript over Typescript. Only early stages of prototyping where fast feedback outweighs the advantages of types and build tools are an exception.
  • React is a power house for building user interfaces. Currently Next.js is the best framework for building TypeScript exclusive web applications. While it's not perfect and nudges you to buy into Vercel's ecosystem, it has a lot of features to increase UX with a DX that is hard to beat.
  • React is not the only way to build user interfaces. While it is still doable with vanilla JavaScript, frameworks help. A lot. It is worth exploring alternatives. Vue and Nuxt are great alternatives to React and Next.js. Nuxt has unqique features like auto imports that I really started to love.
  • If you haven't tried Svelte yet, I can highly recomment it. It's a blast to work with. I did not have chance to try SvelteKit yet so I can't yet compare it to Next.js or Nuxt. But on the client side it is a joy to code with. The svelte compiler does a lot of work for you, but the language stays pretty transparent.
  • The current highlight of my exploration is Elixir, Phoenix and LiveView. The only reason I did not fully commit to it is the learning curve for Elixir and the one downside of Phoenix: It took inspiration from Ruby on Rails, which introduces a lot of indirection. It sticks to MVC instead of more modern patterns that I got used to love. It's a low price to pay for what you get in return: LiveView is a true innovation in web development.
  • I also tried Laravel and Spring Boot. They are great frameworks. They are not for me though. Them sticking to MVC is not the biggest problem. There is too much "magic" in them for my taste and it has real costs. The stack traces are hard to follow. To be honest, Phoenix and React also suffer from this problem, but I found that the advantages of Spring Boot and Laravel are not as compelling compared to React or Phoenix.
  • My current implementation of the application app is written in Golang with Templ for SSR. Moving the application logic to the server side reduces complexity so much that I am not yet missing frontend frameworks. Since most work is done in Go, I do not even use TypeScript. I opted out of SPA though, since the Transition API covers a lot of ground. The difference to a full SPA is hard to notice at least in the development environment. Deployment might break the illusion.
